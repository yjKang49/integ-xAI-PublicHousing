## 1. 목적 및 범위

### 1.1 목적

본 문서는 **AX 공공임대주택 안전유지관리 플랫폼(에이톰-AX)**의 소프트웨어 기능을 체계적으로 정의하고, 각 기능의 입력·처리·출력·접근 권한을 명세한다.  
소프트웨어 조달, 성능 시험, 감리, 유지보수 등에 활용한다.

### 1.2 범위

| 구분 | 포함 범위 |
|------|----------|
| 포함 | 관리자 웹 (admin-web), REST API 서버 (api), AI 워커 (ai-worker), 배경 작업 워커 (job-worker) |
| 제외 | 모바일 앱 (iOS/Android) 클라이언트 측 상세 기능, 인프라 설정 (Nginx, Docker 상세), 외부 AI 모델 내부 알고리즘 |

### 1.3 용어 정의

| 용어 | 정의 |
|------|------|
| KPI | Key Performance Indicator — 핵심 성과 지표 |
| RPA | Robotic Process Automation — 로봇 프로세스 자동화 |
| Human-in-the-Loop | AI 결과를 담당자가 검토·확정하는 절차 |
| XAI | Explainable AI — 설명 가능한 인공지능 |
| 멀티테넌시 | 기관(조직)별로 데이터가 완전 격리되는 SaaS 구조 |
| 게이지 포인트 | 균열 측정 위치를 지정한 고정 관측 지점 |
| 드론 미션 | 드론 비행 및 영상 촬영 임무 단위 |
| Antigravity 엔진 | 비정형 구조물의 의도적 설계 패턴과 실제 손상을 구분하여 오탐(False Positive)을 원천 차단하는 보정 엔진 |
| Y-MaskNet | Instance Segmentation 기반 결함 탐지 비전 AI 모델 (에이톰엔지니어링 개발) |
| KDS | Korean Design Standard — 국가 건설기준(설계기준). 점검 보고서 법적 근거 기준 |
| KALIS-FMS | 국토안전관리원 시설물통합정보관리시스템 — 과거 결함 이력 빅데이터 API |
| 세움터 | 국토교통부 건축물대장 포털 — 구조 형식·설계 하중·지반 정보 API |
| LIO-SLAM | LiDAR-Inertial Odometry SLAM — GPS 음영지역에서 6방향 센서 융합 3D 점군 맵핑 기술 |
| 드라이비트 | 공공주택 외벽에 사용되는 가연성 단열재(DRYVIT). 화재 취약 구조의 주요 위험 요소 |
| On-Premise | 현장 폐쇄망 내 국산 NPU 기반 엣지 서버 구축 방식. 데이터가 외부망으로 이탈하지 않음 |

## 3. 시스템 개요

### 3.1 시스템 구성

**클라우드/SaaS 모드** (기본 구성):

```
┌─────────────────────────────────────────────────────────┐
│                  사용자 (브라우저/모바일)                  │
└────────────────────┬────────────────────────────────────┘
                     │ HTTPS
┌────────────────────▼────────────────────────────────────┐
│              Nginx 리버스 프록시 (SSL 종단)               │
└──────┬──────────────────────────┬───────────────────────┘
       │                          │
┌──────▼──────┐          ┌────────▼──────┐
│  admin-web  │          │   API 서버    │
│ Angular 18  │          │  NestJS 10    │
│  (정적 SPA) │          │  (REST API)   │
└─────────────┘          └───┬──────┬───┘
                             │      │
              ┌──────────────┘      └──────────────┐
              │                                    │
    ┌─────────▼──────────┐              ┌──────────▼────────┐
    │  데이터 계층        │              │  작업 계층         │
    │  CouchDB 3.3 (DB)  │              │  Redis 7.2         │
    │  Redis 7.2 (캐시)  │              │  Bull Queue        │
    │  MinIO/S3 (파일)   │              │  ai-worker         │
    └─────────────────────┘              │  job-worker        │
                                         └────────────────────┘
```

**On-Premise(폐쇄망) 모드** — 공공기관 현장 배포 시:

```
┌──────────────────────────────────────────────────────────────┐
│           현장 폐쇄망 (외부망 차단, 개인정보 외부 유출 없음)  │
│                                                              │
│  ┌────────────────────────┐   ┌───────────────────────────┐  │
│  │  국산 NPU 엣지 서버     │   │  API 서버 + 데이터 계층   │  │
│  │  (AI 추론 전용)         │   │  (동일 구성, 내부망만)    │  │
│  │  · Y-MaskNet 추론       │   │  · CouchDB / Redis        │  │
│  │  · Antigravity 보정     │   │  · MinIO (로컬 S3)        │  │
│  │  · OpenCV 균열 분석     │   │  · Bull Queue 워커        │  │
│  │  · 드론 영상 비식별화   │   └───────────────────────────┘  │
│  └────────────────────────┘                                  │
│                                                              │
│  외부 연동: KALIS-FMS API, 세움터 API (정부 전용망 경유)      │
└──────────────────────────────────────────────────────────────┘
```

> **보안 원칙:** On-Premise 모드에서는 드론·앱으로 수집된 입주민 개인정보(얼굴, 차량번호 등)가 국산 NPU 서버에서 실시간 비식별화 처리되며 외부로 이탈하지 않는다. 「개인정보보호법」 및 국가정보보안지침 준수.

### 5.1 인증 및 사용자 관리

#### F-AUTH-001 | 로그인

| 항목 | 내용 |
|------|------|
| 기능명 | 사용자 로그인 |
| 설명 | 이메일·비밀번호 입력으로 JWT Access Token(15분) 및 Refresh Token(7일)을 발급한다. |
| 입력 | 이메일(string), 비밀번호(string) |
| 처리 | bcrypt 해시 비교 → JWT 생성 → Refresh Token Redis 저장 |
| 출력 | `{ accessToken, refreshToken, user: { id, name, role, organizationId } }` |
| 접근 권한 | 모든 사용자 (비인증 허용) |
| 관련 API | POST /auth/login |
| 우선순위 | P1 |

#### F-AUTH-002 | 토큰 갱신

| 항목 | 내용 |
|------|------|
| 기능명 | Access Token 자동 갱신 |
| 설명 | Refresh Token으로 새로운 Access Token을 발급한다. Refresh Token이 Redis deny-list에 있으면 거부한다. |
| 입력 | Refresh Token (Bearer) |
| 처리 | Redis deny-list 확인 → JWT 검증 → 새 Access Token 생성 |
| 출력 | `{ accessToken }` |
| 접근 권한 | Refresh Token 보유자 |
| 관련 API | POST /auth/refresh |
| 우선순위 | P1 |

#### F-AUTH-003 | 로그아웃

| 항목 | 내용 |
|------|------|
| 기능명 | 로그아웃 (토큰 무효화) |
| 설명 | Refresh Token을 Redis deny-list에 등록하여 즉시 무효화한다. |
| 입력 | Refresh Token (Bearer) |
| 처리 | Redis에 토큰 해시 등록 (TTL = 잔여 유효기간) |
| 출력 | 204 No Content |
| 접근 권한 | 인증된 사용자 |
| 관련 API | POST /auth/logout |
| 우선순위 | P1 |

#### F-AUTH-004 | 사용자 생성

| 항목 | 내용 |
|------|------|
| 기능명 | 사용자 계정 생성 |
| 설명 | 조직 내 새 사용자를 생성하고 역할(Role)을 배정한다. |
| 입력 | 이름, 이메일, 초기 비밀번호, 역할, 소속 단지(선택) |
| 처리 | 이메일 중복 확인 → bcrypt 해싱 → CouchDB 저장 |
| 출력 | 생성된 사용자 정보 (비밀번호 제외) |
| 접근 권한 | SUPER_ADMIN, ORG_ADMIN |
| 관련 API | POST /users |
| 우선순위 | P1 |

#### F-AUTH-005 | 사용자 관리 (조회·수정·삭제)

| 항목 | 내용 |
|------|------|
| 기능명 | 사용자 목록 조회 / 수정 / 비활성화 |
| 설명 | 소속 기관 내 사용자 목록 조회, 정보 수정, Soft Delete(isActive=false) 처리. |
| 입력 | (조회) 페이지·검색어 / (수정) 이름·역할·배정 단지 / (삭제) userId |
| 처리 | 조직별 DB 격리 조회 → CouchDB CRUD |
| 출력 | 사용자 배열 or 수정된 사용자 객체 |
| 접근 권한 | SUPER_ADMIN, ORG_ADMIN |
| 관련 API | GET/PATCH/DELETE /users, GET /users/:id |
| 우선순위 | P1 |

### 5.2 시설물 계층 관리

#### F-FAC-001 | 단지 관리

| 항목 | 내용 |
|------|------|
| 기능명 | 공동주택 단지 등록·조회·수정·삭제 |
| 설명 | 단지(Complex)를 등록하고, 단지에 속한 건물·층·구역을 계층적으로 관리한다. QR 코드 자동 생성 포함. |
| 입력 | 단지명, 주소, 총 세대수, 건물 수, 준공연도, 위도/경도(선택), 평면도 파일(선택) |

| 500 | AX-5001 | 내부 서버 오류 | 예상치 못한 예외 |
| 502 | AX-5021 | AI 워커 응답 없음 | Bull Queue Job 타임아웃 |
| 503 | AX-5031 | 서비스 불가 | CouchDB/Redis 연결 실패 |

### 11.2 장애 유형별 대응 전략

| 장애 유형 | 감지 방법 | 자동 대응 | 수동 대응 | 목표 복구 시간 |
|---------|---------|---------|---------|------------|
| API 서버 다운 | Docker healthcheck (30초 간격) | restart=always 자동 재시작 | docker compose up ax-api | ≤ 30초 |
| CouchDB 연결 실패 | NestJS ConnectionPool 모니터링 | 연결 재시도 (3회, 지수 백오프) | docker restart ax-couchdb | ≤ 60초 |
| Redis 연결 실패 | Bull Queue 연결 감시 | 캐시 우회 (DB 직접 조회) | docker restart ax-redis | ≤ 30초 |
| Bull Queue 작업 실패 | attempts=3 초과 시 failed 상태 | 자동 재시도 (3회, 5초 간격) | 수동 Job 재실행 또는 삭제 | ≤ 3분 |
| MinIO 연결 실패 | Pre-signed URL 생성 실패 감지 | HTTP 503 반환, 업로드 차단 | docker restart ax-minio | ≤ 60초 |
| AI 워커 다운 | Job 처리 타임아웃 (60초) | Job FAILED 상태 전이 | docker restart ax-ai-worker | ≤ 30초 |
| 메모리 부족 | Container stats 임계치 초과 | OOM Killer 종료 후 재시작 | 메모리 한도 조정 및 재시작 | ≤ 5분 |
| NAS 연결 끊김 | 파일 읽기 ENOENT 에러 | 오류 반환 (AX-5001) | NAS 연결 복구 후 재시도 | 수동 |

### 11.3 AI 파이프라인 장애 대응

### 12.3 KALIS-FMS 및 세움터 연동 요구사항

| 외부 시스템 | 연동 방식 | 데이터 항목 | 연동 주기 |
|----------|---------|---------|---------|
| KALIS-FMS (한국시설안전공단) | REST API (JSON) | 결함 이력, 점검 결과, 위험도 점수 | 점검 세션 완료 시 즉시 |
| 세움터 (국토교통부) | REST API (JSON) | 건물 정보, 점검 이력, 균열 측정값 | 월 1회 배치 동기화 |

**연동 오류 대응:**
- KALIS-FMS API 응답 없음: 재시도 3회 후 pending 상태 보관, 다음 배치 재처리
- 세움터 인증 만료: 토큰 갱신 후 자동 재처리
- 데이터 형식 불일치: 변환 오류 로그 기록, SUPER_ADMIN 알림
