> **작성일:** 2026-04-14  
> **상태:** Phase 2 종료 시점 기준

## 1. 기술적 리스크

### RISK-T01 — AI 모델 Mock 구현
- **등급:** HIGH
- **내용:** 결함 탐지(DEFECT_DETECTION), 균열 분석(CRACK_ANALYSIS) 프로세서가 실제 모델 대신 Mock 응답을 반환함. 프로덕션 배포 전 실제 모델 통합 필요.
- **영향:** 데모는 정상 작동하나, 실 데이터로 정확도 검증 불가
- **완화 방안:** Feature Flag(`AI_DEFECT_DETECTION`, `AI_CRACK_ANALYSIS`)로 격리됨. 실증 전에 실제 모델 교체 필수.
- **Phase 3 조치:** 실제 Mask R-CNN / Y-MaskNet 모델 서빙 (TorchServe / ONNX Runtime)

### RISK-T02 — LLM API 비용 및 속도 불확실성
- **등급:** MEDIUM
- **내용:** Claude API 호출은 외부 의존성으로, 응답 시간 변동(3~15초)과 비용이 사용량에 따라 변함.
- **영향:** 민원 급증 시 AI 분류 지연 가능, 운영 비용 예측 어려움
- **완화 방안:** 잡 큐로 비동기 처리, `RPA_DRY_RUN` 플래그로 비용 제어
- **Phase 3 조치:** 응답 캐싱, 온프레미스 LLM 옵션 검토 (llama.cpp / vLLM)

### RISK-T03 — 실시간 센서 스트리밍 미구현
- **등급:** MEDIUM
- **내용:** 현재 센서 데이터는 REST API polling 방식. WebSocket/MQTT 실시간 스트리밍 미구현.
- **영향:** 경보 발생이 polling 주기(현재 미정)에 의존 — 실시간 경보 불가
- **완화 방안:** 임계치 초과 감지는 API 호출 시 즉시 처리
- **Phase 3 조치:** MQTT 브로커 연동 또는 Server-Sent Events 구현

### RISK-T04 — CouchDB 대규모 데이터 성능
- **등급:** MEDIUM
- **내용:** 단일 CouchDB 인스턴스로 수십만 문서 처리 시 Mango 쿼리 성능 저하 가능.
- **영향:** 대형 기관(단지 수 100개 이상) 적용 시 응답 지연
- **완화 방안:** 현재 인덱스 25개 정의, 복합 인덱스 최적화 완료
- **Phase 3 조치:** CouchDB 클러스터링 또는 Elasticsearch 집계 레이어 추가

### RISK-T05 — 균열 측정 보정 기준 미표준화
- **등급:** MEDIUM
- **내용:** 균열 폭 측정 시 픽셀-밀리미터 보정에 카메라 거리 정보가 필요하나, 드론 고도 데이터 연동이 완전하지 않음.
- **영향:** 측정값 오차 증가 (원거리 촬영 시 ±0.5mm 이상 가능)
- **완화 방안:** 게이지 포인트에 물리적 기준자 배치 권장
- **Phase 3 조치:** 드론 고도·자세 데이터 연동, 보정 알고리즘 표준화

## 2. 운영적 리스크

### RISK-O01 — 이메일/SMS 실발송 미구현
- **등급:** LOW (파일럿), HIGH (본사업)
- **내용:** 알림 발송이 Mock 구현(`RPA_DRY_RUN = true`). 실제 이메일/SMS 발송 미구현.
- **영향:** 파일럿 중 실제 알림 전달 불가, 사용자 경험 제한
- **Phase 3 조치:** SendGrid/AWS SES 이메일, Twilio/KakaoTalk 알림 연동

### RISK-O02 — 모바일 앱 미배포
- **등급:** MEDIUM
- **내용:** Ionic 앱 코드는 존재하나 iOS/Android 스토어 배포 미진행.
- **영향:** 현장 점검자가 웹 브라우저로만 접근 가능
- **Phase 3 조치:** TestFlight + Google Play 내부 배포 → 스토어 정식 출시

### RISK-O03 — 백업 자동화 미구현
- **등급:** MEDIUM
- **내용:** CouchDB 정기 백업 자동화 스크립트 미구현. 현재 수동 백업 의존.
- **영향:** 장애 발생 시 데이터 유실 가능
- **Phase 3 조치:** 일간 자동 백업 크론잡 + S3/MinIO 오프사이트 백업

### RISK-O04 — 다중 기관 동시 운영 미검증
- **등급:** MEDIUM
- **내용:** 멀티테넌트 설계는 완료되었으나, 10개 이상 기관 동시 접속 부하 테스트 미실시.
- **영향:** 대규모 파일럿 전개 시 성능 병목 예측 어려움
- **Phase 3 조치:** 부하 테스트 (k6/Artillery) 실시, 성능 튜닝

## 3. 보안 리스크

### RISK-S01 — API 키 환경변수 관리
- **등급:** MEDIUM
- **내용:** Claude API Key, CouchDB 패스워드 등이 `.env` 파일에 평문 저장.
- **완화 방안:** `.env` 파일은 `.gitignore` 처리, 시드 코드에 하드코딩 없음
- **Phase 3 조치:** AWS Secrets Manager 또는 HashiCorp Vault 도입

### RISK-S02 — Rate Limiting 세분화 부족
- **등급:** LOW
- **내용:** API Rate Limit이 전체 단위로 설정. AI 엔드포인트에 별도 제한 미설정.
- **완화 방안:** NestJS Throttler 기본값 적용 중
- **Phase 3 조치:** AI 엔드포인트별 세분화된 Rate Limit + DDoS 방어

## 5. 기술 부채

| 항목 | 위치 | 내용 |
|------|------|------|
| `strictNullChecks: false` | tsconfig.json | 타입 안전성 향상 필요 |
| `noImplicitAny: false` | tsconfig.json | any 타입 사용 허용 중 |
| 일부 `any` 캐스팅 | automation-rules.service.ts | 타입 명시 개선 필요 |
| 단위 테스트 커버리지 | apps/api/src/ | 현재 E2E 위주, 단위 테스트 부족 |
| 에러 코드 표준화 | 전체 API | HTTP 오류 코드가 일부 비일관적 |
