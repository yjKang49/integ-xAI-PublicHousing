# AI 모델 거버넌스 개요

> **목적:** Phase 2 AI 시스템의 책임 있는 운영을 위한 거버넌스 체계 정의  
> **기준 근거:** AX-SPRINT 국가R&D 과제 AI 책임성 요건, OECD AI 원칙

---

## 1. 거버넌스 원칙

### 1.1 Human-in-the-Loop 원칙

모든 AI 결과는 자동 확정되지 않으며, 다음 두 단계 승인 구조를 따른다.

```
AI 결과 생성 (DRAFT)
     ↓
담당자 검토 (REVIEWING)
     ↓
승인 또는 반려 (APPROVED / REJECTED)
```

| AI 기능 | 자동 확정 가능 조건 | 승인 권한 |
|---------|---------------------|-----------|
| 결함 탐지 | AI_AUTO_ACCEPT 플래그 ON + confidence ≥ 0.90 | INSPECTOR 이상 |
| 균열 분석 | 불가 (항상 검토 필요) | REVIEWER 이상 |
| 진단 의견 | 불가 (항상 DRAFT → 승인) | REVIEWER 이상 |
| 민원 분류 | 불가 (배정 추천만, 확정은 담당자) | COMPLAINT_MGR 이상 |

### 1.2 투명성 원칙

- AI 판단 근거는 항상 사용자에게 표시 (설명가능 AI)
- 모델 버전, 신뢰도, 처리 일시를 결과에 기록
- 승인/반려 이력과 담당자를 영구 보존

### 1.3 공정성 원칙

- 특정 건물 또는 지역에 대한 편향 없이 동일 기준 적용
- 분기별 카테고리별 탐지 비율 검토로 편향 모니터링

## 2. 역할 및 책임 (RACI)

| 활동 | AI 운영팀 | 구조공학 전문가 | 시스템 관리자 | 기관 담당자 |
|------|-----------|----------------|--------------|-------------|
| 모델 성능 모니터링 | **R** | C | I | I |
| 정기 샘플 검토 | A | **R** | I | C |
| 모델 재훈련 결정 | **R** | A | C | I |
| 거부율 이상 조사 | **R** | C | I | I |
| Feature Flag 변경 | A | I | **R** | I |
| 사고 보고 | **R** | C | A | I |

> R: Responsible, A: Accountable, C: Consulted, I: Informed

## 3. AI 사용 이력 관리

### 3.1 보존 항목

모든 AI 생성 결과에 다음 메타데이터를 기록한다.

```typescript
// 공통 AI 이력 필드
interface AiAuditFields {
  aiGeneratedAt: string;      // ISO timestamp
  aiModelVersion: string;     // e.g., "defect-detection-v2.1"
  aiConfidence: number;       // 0~1
  reviewedBy?: string;        // userId (승인자)
  reviewedAt?: string;        // ISO timestamp
  reviewDecision: 'APPROVED' | 'REJECTED' | 'PENDING';
  reviewComment?: string;     // 반려 사유 등
}
```

### 3.2 보존 기간

| 데이터 유형 | 보존 기간 | 근거 |
|-------------|-----------|------|
| AI 탐지 결과 | 10년 | 시설물 관리 법정 기준 |
| 진단 의견 이력 | 10년 | 구조물 안전관리법 |
| 민원 분류 이력 | 5년 | 공공기록물 관리 |
| 모델 평가 보고서 | 영구 | R&D 과제 산출물 |

## 4. 사고 대응 절차

### 4.1 사고 등급 분류

| 등급 | 정의 | 예시 | 대응 시간 |
|------|------|------|-----------|
| P1 (긴급) | AI 오탐으로 안전사고 위험 | 균열 없는 건물에 CRITICAL 판정 | 즉시 (1시간 이내) |
| P2 (경고) | 대규모 오탐 발생 | 특정 이미지 배치 전체 오탐 | 4시간 이내 |
| P3 (주의) | 성능 지표 임계치 초과 | 월별 거부율 15%p 증가 | 다음 영업일 |

### 4.2 P1 긴급 대응

```
1. Feature Flag 즉시 비활성화
   → PATCH /api/v1/feature-flags/{key}  enabled: false

2. 영향 범위 확인
   → 해당 날짜 AI 생성 결과 전체 재검토 요청

3. 원인 분석 및 보고서 작성
   → 48시간 이내 사후 분석 보고서 제출

4. 재활성화 승인
   → 구조공학 전문가 + 시스템 관리자 공동 승인
```

## 5. 모델 생명주기 관리

```
데이터 수집 → 모델 훈련 → 검증 (≥기준 충족) → 스테이징 배포
     ↑                                               ↓
 피드백 수집                                   A/B 테스트 (선택)
     ↑                                               ↓
모니터링 ← 프로덕션 운영 ← Feature Flag 점진적 활성화
```

### 5.1 모델 업데이트 기준

| 트리거 | 조건 | 조치 |
|--------|------|------|
| 성능 저하 | mAP 기준 -0.05 이상 | 즉시 재훈련 검토 |
| 새 결함 유형 | 현장 신규 패턴 발생 | 데이터 수집 후 파인튜닝 |
| 법규 개정 | KCS 기준 변경 | 라벨링 기준 재정의 |
| 정기 업그레이드 | 연 1회 | 전체 재평가 후 배포 |
